# 缓存
在计算机的世界里「缓存」这个词的占比非常重。无论是在 CPU、浏览器、服务器，或者是其他环境、设备上，都会涉及到缓存。缓存的核心逻辑其实就是一套「局部性」原理。

以此类推，我们在大模型上，也会应用「缓存」。缓存一般也有三种情况，一种是精确匹配，另一种是前缀缓存，第三种是语义缓存。每种情况都有各自的优点和缺点吧。

## KV 缓存（精确匹配）
一些简单的推理框架使用的是 KV 缓存（Key-Value Cache），顾名思义就是用户输入的问题进行一个完整的 Hash 处理。

当我们第二个问题和第一个问题只要有一些不一样（比如标点符号，或者换了一个表述方式）那这个缓存就无法再使用了。

它的缺点就是说太生硬了，每次都会进行大量的重新计算。

## 前缀缓存
当大模型在处理文本的时候，会将文本切成一个一个的token缓存不是针对字符串的，而是针对整个token序列的。如果两个问题开头的部分是完全一致的，你比如说都引用了相同的文档，那么框架会缓存这部分token的计算结果。

换句话说，如果你改变了表达方式，开头的语义完全不一样了，那前置缓存其实也无法命中的。

目前全缀缓存用的还是比较多的，如果两个请求在开头的部分完全一致，那么系统就会跳过这部分计算，直接从内存或者显存中读之前的这个计算状态。这个大部分情况下还是针对于系统提示词以及用户提示词，或者是一些引用的长文档的内容。在这种情况下，他们前面的那些数据是一致的，所以可以把它们缓存出来，不会有任何问题。

## 语义缓存
有的时候我们提出两个问题，他最终想要的结果是一样的，只不过它表达方式有一些区别。在这种情况下，我们就需要引入语义缓存。当我们输入第一个问题的时候，系统不只是把它当做字符串，而是通过一个其他的一个小的模型将它转换成一个向量值。

将这个向量值还有模型生成的这个答案缓存起来。

当我们在问第二一个问题的时候，也把第2一个问题的问题做成向量，然后去和我们缓存的问题去做余弦相似度匹配。比如相似度如果超过某一个我们设定的临界值，那就可以判断这两个问题的语义是一致的。

当我们判断语义是一致的情况下，就可以直接返回上一次缓存的答案。

橘黄色也有自己相应的挑战，比如说帮我写一个关于猫的故事和帮我写一个关于猫悲伤的故事，虽然语意很像，但它们的答案其实一定是不一样的。或者是在多轮对话里面，上下文是不断变化的，同一个问题，在不同的对话阶段，它所参考的背景资料也是不一样的，所以它给到的答案也应该是不同的。

在我们使用中，我们可以在前端做一层语义缓存，只不过要把这个语义缓存的这个相似度提到一个非常高的值，比如说0.98这种。如果匹配成功，就可以直接返回历史答案，不会重新去生成一些数据。

然后我们还要做一些前缀匹配，省去了这些背景的计算开销。你只要改变你的这个表达方式，都会当成新问题去处理。


## 如何缓存数据？
我们在与大模型沟通的时候，当第一次问题和第二次的问题，语义一样时，那么第二次就会使用第一次缓存的结果。

那 ... 如何判断第一次和第二次问题语义是一致的呢？
我们可以使用一个「embedding」库，把第一次提出的问题和第二次提出的问题都转成向量值，通过「余弦」计算逻辑去比较它们的差异。

差异的对比会得到一个数字值，如果值小于我们设定的差异值，则说明两个问题意图是一致的，我们就可以直接输出上一次缓存答案。如果这个结果值小于我们设置的差异值，那说明意图不一致，需要重新去生成结果。

## nomic-embed-text
它是一个文本向量化模型（embedding model），它只负责把文本转换成数字向量，它就这一个功能。

一般情况下，我们可以用它来做语义检索、知识库检索、文本归纳等等工程。

### 知识库（RAG）典型流程是：
- 用 nomic-embed-text 把文档切片后转成向量
- 把生成的向量值存进向量数据库
- 用户提问时，把问题转成向量
- 找最相近的文档片段
- 再交给聊天模型生成答案

我们今天做一个简单的实验，用来理解这个模型在解决什么问题，我本地部署了 Ollama，使用的是 Qwen3.5:9B 的小模型。我向模型提出两个问题，这两个问题的语义是一致的，看它是否会给我返回相同的结果。

## 示例
第一个问题
```
请你介绍一下LLM?简述,100字以内。
```
大模型给出的回复，如下：
```
LLM 指大型语言模型，基于海量数据训练的深度神经网络。它能理解并生成自然语言，广泛应用于文本生成、翻译、问答等场景，推动人工智能在各领域的创新应用。
```
第二个问题
```
说一下LLM是什么?
```

我这两个问题的含义是一致的，我都在是问「LLM 是什么？」，因为大模型只能接收数字，它只能靠向量值去理解问题的含义。


最一开始我没配置「向量化模型」匹配，所以，我第二次问出相同含义的问题时，大模型重新进行了计算，两次生成的结果是不同的。

### 配置文本转向量
于是我开始配置文本转向量，每次输入问题都把问题转成向量存储，并且把推理结果也一并存储到 cache 里，缓存起来。
```
import { Ollama, OllamaEmbeddings } from "@langchain/ollama"
// 余弦比较
import cosineSimilarity from 'cosine-similarity';

// 用来回答问题
const model = new Ollama({
    model: "qwen3.5:9b",
    streaming: true,
    think: false
})

// 用来文本转换成向量
const embeddings = new OllamaEmbeddings({
    model: "nomic-embed-text"
})

const cache = [];
async function callLLM(inputText, threshold = 0.7) {
    // 计算输入文本的向量
    const queryEmbedding = await embeddings.embedQuery(inputText);

    // 计算输入文本的向量与缓存中每个向量的余弦相似度
    for (const item of cache) {
        const similarity = cosineSimilarity(queryEmbedding, item.embedding);
        if (similarity >= threshold) {
            console.log("缓存命中 ... 相似度：", similarity, "\n\n");
            // 缓存命中
            process.stdout.write(item.response + "\n");
            return;
        }
    }

    // 缓存未命中
    console.log("缓存没命中 ... \n\n");
    let fullRes = "";
    const stream = await model.stream(inputText);
    for await (const chunk of stream) {
        process.stdout.write(chunk);
        fullRes += chunk;
    }

    // 4. 存储到缓存里面
    cache.push({
        input: inputText, // 用户原始的提示词
        embedding: queryEmbedding, // 提示词对应的向量
        response: fullRes, // 模型给的回复
    });
}
```
所以，最终即使这两个文本相差的字数特别多，但是通过向量比较分析，它们能达到 70% 以上的相同语意的，可以看一下我最终的输出结果。
```
缓存命中，相似度： 0.7297206609621
```
然后把上面缓存的结果，一次性回复了出来。
```
LLM 指大型语言模型，基于海量数据训练的深度神经网络。它能理解并生成自然语言，广泛应用于文本生成、翻译、问答等场景，推动人工智能在各领域的创新应用。
```

## 现状
对于大模型厂商来说，他们一般会默认做前缀缓存，比如说你依赖的几千字的文档，或者是说你用户的这个系统提示词、用户级别的提示词，他们都会自己做前缀缓存。
那如果我们还想提高这个命中率的话，我们可以在前端做一层语义缓存，就是把缓存匹配值加到一个非常高的值，尽量不影响用户的体验。
这种高分值的策略，比如说0.95~0.98，这种意图是极度的接近了，这样可以保证正确率，又过滤掉一些重复请求。
